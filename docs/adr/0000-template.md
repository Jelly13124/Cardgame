## Context
What problem are we solving? What constraints exist (technical, project, team, time)? What forced this decision now?

Keep it tight — 1-3 short paragraphs. The reader needs enough to evaluate the alternatives, not a system overview.

## Decision
The actual choice, stated plainly. Often a single paragraph.

If the decision has multiple parts (e.g. "X via Y, with Z fallback"), list them as bullets.

## Alternatives Considered
This is the most important section. For each viable alternative:

### Alternative 1: Name
- **Pros:** why this looked attractive
- **Cons:** why we didn't pick it
- **Why rejected:** the deciding factor

### Alternative 2: Name
- **Pros:** ...
- **Cons:** ...
- **Why rejected:** ...

(2-4 alternatives is typical. Including "do nothing" is often valuable.)

## Consequences

**Positive:**
- What gets easier / better as a result

**Negative / Trade-offs:**
- What we gave up
- What pain we're accepting

**Risks (and mitigations):**
- What could go wrong with this choice
- What signal would tell us to revisit

## Revisit Triggers
Conditions under which we should re-open this decision:
- e.g. "If we add multiplayer"
- e.g. "If team grows past 3 people"
- e.g. "If JSON file count exceeds 500"
